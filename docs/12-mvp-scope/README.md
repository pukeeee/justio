# Розділ 12: Визначення рамок MVP (Minimum Viable Product)

Мета MVP — швидко випустити на ринок робочий продукт, який надає ключову цінність цільовій аудиторії та дозволяє зібрати реальний фідбек. Все, що не є абсолютно необхідним для цього, відкладається на наступні ітерації.

## 12.1. Що Входить до MVP

### Керування Користувачами та Організацією
*   **Реєстрація та Вхід:** Реєстрація нового користувача з автоматичним створенням організації. Вхід за допомогою email + пароль або magic link.
*   **Базове керування командою:** Адміністратор може запросити в організацію **одного** іншого користувача з роллю "Менеджер".
*   **Налаштування профілю:** Можливість змінити назву організації.

### Ядро CRM (Клієнти та Угоди)
*   **Клієнти (CRUD):** Повноцінне створення, редагування та видалення (м'яке) контактів та компаній.
*   **Список клієнтів:** Перегляд списку клієнтів з можливістю пошуку за іменем.
*   **Угоди (CRUD):**
    *   Створення угоди з прив'язкою до існуючого клієнта.
    *   Можливість додати до угоди кілька позицій (товарів/послуг) з назвою, кількістю та ціною. Для цього буде використано UI-патерн **"повторювана група полів"** (repeating form group).
    *   Перегляд списку угод з фільтрацією за статусом.
    *   Зміна статусу угоди (наприклад, "Нова" -> "В роботі" -> "Успішна").
    *   Видалення угод у статусі "чернетка".

### Інтеграції (мінімальна реалізація)
*   **Інтеграція з «Нова Пошта»:**
    *   На сторінці угоди є кнопка "Створити ТТН".
    *   При натисканні викликається API Нової Пошти. Для спрощення в MVP може використовуватися єдиний тестовий ключ для всіх користувачів, а не індивідуальні ключі кожної організації. Номер ТТН зберігається в угоді.
    *   **Не реалізовано в MVP:** автоматичне заповнення адрес, відстеження статусів.
*   **Email-сповіщення:**
    *   Кнопка "Надіслати email клієнту" на сторінці угоди, яка відправляє шаблонний лист (наприклад, "Ваше замовлення #123 відправлено").
    *   **Не реалізовано в MVP:** редактор шаблонів, масові розсилки.

### UX/UI та Офлайн-режим
*   **Адаптивний дизайн:** Ключові сторінки (списки, форми) коректно відображаються на мобільних пристроях.
*   **PWA:** Додаток можна "встановити" на головний екран.
*   **Базовий офлайн-режим:**
    *   Кешування статичних ресурсів (додаток відкривається офлайн).
    *   Можливість **переглядати** раніше завантажені дані (клієнтів, угоди).
    *   **Частково реалізовано:** створення нових записів офлайн з подальшою синхронізацією (для однієї сутності, наприклад, клієнтів, для демонстрації концепції).

### Підписка та Білінг (імітація)
*   **Без фактичної оплати:** В MVP не буде інтеграції з реальними платіжними шлюзами.
*   **Імітація тарифів:** В базі даних є поле "тариф". Для симуляції переходу на інший план в налаштуваннях адміністратора буде додано **тестовий перемикач (dummy toggle)**, який змінюватиме тариф в БД.
*   **Застосування лімітів:** Реалізовано одне просте обмеження (наприклад, не більше 10 клієнтів на безкоштовному тарифі), щоб продемонструвати логіку.
*   **Тріальний період:** Логіка автоматичного завершення тріального періоду не реалізується; за потреби, цей процес може контролюватися вручну.

## 12.2. Що НЕ Входить до MVP

*   **Портал для постачальників (B2B2B):** Вся логіка, пов'язана з іншими організаціями.
*   **Просунуті модулі:** Складський облік, поглиблена аналітика з графіками, бухгалтерські інтеграції.
*   **Відкритий API та Marketplace:** Можливість для сторонніх розробників інтегруватися з нашою системою.
*   **Багатомовність:** Інтерфейс буде реалізовано лише українською мовою.
*   **White-label:** Кастомізація брендингу та використання власного домену.
*   **Повноцінний офлайн-режим:** Складне вирішення конфліктів, синхронізація всіх типів даних та файлів.
*   **Автоматизація (Workflows):** Налаштування ланцюжків автоматичних дій.
*   **Розширені налаштування прав доступу:** Створення кастомних ролей, дозволи на рівні полів.
*   **Автоматизоване тестування та CI/CD:** В рамках MVP розробка може покладатися на ручне тестування.
*   **Система підтримки:** Вбудований чат підтримки, детальна документація для користувачів.
*   **Імпорт/Експорт даних:** Масове завантаження та вивантаження даних (може бути реалізовано на рівні SQL-запитів для внутрішніх потреб, але не в UI).

## 12.3. Вимоги до якості MVP
Незважаючи на обмежений функціонал, MVP має бути стабільним та надійним у межах заявлених можливостей. Перед запуском для перших користувачів необхідно провести **наскрізне ручне тестування** ключових сценаріїв:
1.  Реєстрація нового користувача та його організації.
2.  Створення клієнта.
3.  Створення угоди для цього клієнта.
4.  Зміна статусу угоди.
5.  (Якщо реалізовано) Генерація ТТН для угоди.

Усі ці кроки повинні виконуватися без критичних помилок. Функціонал, що не входить до MVP, має бути або прихованим, або деактивованим, щоб не вводити користувачів в оману.
